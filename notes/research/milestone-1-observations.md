# Milestone 1 Observations

- Launcher probe status: `captures/20260418-154355-runner/launcher-http.log` recorded `GET /rss` and `GET /updatev2/MiniUpdate/update.info` from the stock launcher. `HttpProbeService` answered with `SERVER ONLINE!` and `NO_UPDATE`, and the launcher proceeded to start `Game.exe`.
- Server list status: decoded the stock `SvrList.dta` and `SvrListM.dta` formats into structured rows with `emulator.tools.server_list`. The classic login entry resolves to `GLOBAL 79.137.101.33 4005`, and the menu server list rewrites cleanly to a localhost endpoint while preserving the original row shape.
- Login gateway status: the first localhost-captured login packet is `captures/20260418-162948-runner/packets/conn-0001-in-0001.bin` at 74 bytes. Its first 16 bytes are `1100b10b7a975ec71045e61875b433ab`, and the exact request was saved to `notes/research/login-script.json`.
- Scripted replay status: the initial 4-byte stub reply `90000000` was implemented and replayed successfully at the transport layer, but the client did not advance with that minimal response alone. The real remote failure reply was then captured as `notes/research/login-failure-script.json`, and the replay engine gained `close_after_response` support so failure scripts can mimic the upstream disconnect behavior exactly.
- Proxy gateway status: `LoginGatewayService` now supports upstream proxy mode in addition to raw capture and scripted replay. The successful live capture run used `127.0.0.1:14021 -> 79.137.101.33:4005` while preserving every inbound and outbound packet locally under `captures/20260418-162948-runner`.
- Live auth outcome: the proxied localhost login path now works end-to-end. The user confirmed that they were able to log in and create a character from the stock client against the localhost shim.
- Packet notes: the successful proxied run expanded to 15 inbound packets and 7 outbound packets. Notable replies include `conn-0001-out-0002.bin` at 50 bytes, `conn-0001-out-0004.bin` at 28 bytes with an ASCII fragment containing `check 0`, `conn-0001-out-0006.bin` at 182 bytes, and `conn-0001-out-0007.bin` at 2737 bytes. Plaintext fragments observed in proxied server traffic include `art`, `33114`, `check 0`, and `OK`.
- Client startup pitfall: the `Unable to read skill information.` popup was not a server failure. Ghidra traced it to `FUN_0076d990`, which opens relative files such as `data/menu/str/US.txt`; launching `Game.exe` without the client directory as the working directory causes that read to fail even when the file exists.
- Extra service forwarders: ad-hoc localhost proxies for ports `4007` and `4008` were also run and logged under `captures/20260418-165923-proxy-4007` and `captures/20260418-165923-proxy-4008` so later milestones can inspect post-login service traffic separately from the main login gateway.
- Unified runner outcome: the built-in multi-port runner in `captures/20260418-172702-runner` now covers the login proxy plus optional `4007` and `4008` listeners in one command. The final successful live run grew to 159 packet captures on the main `14021` connection, confirming the client can remain active in-world through the localhost shim without spinning up separate packet folders for `4007` or `4008`.
- World-entry outcome: after logging in, selecting a character, and clicking Play, the stock client reached the live game world through the localhost proxy. The user reported and showed an in-world scene with their character loaded, which confirms the current stack can drive launcher, login, character selection, and initial world entry against the upstream server.
- Upstream-content note: the in-world scene still looks like the official server because that is exactly what the current emulator is doing. The localhost stack is acting as a transparent protocol capture shim in front of the real upstream service, not yet a standalone world emulator.
- GPU crash root cause: repeated end-of-load crashes were not caused by bad protocol handoff packets. Windows Error Reporting consistently logged `Game.exe` failing with `0xc0000409`, most recently in `igd10um32xe.DLL` with fault offset `0x006c6e23`, pointing to the Intel graphics driver path during world load.
- GPU mitigation that worked: forcing `C:\Users\skull\OneDrive\Documents\savage eden\client\LaghaimOnlineNew\Game.exe` to use the high-performance GPU via `HKCU\Software\Microsoft\DirectX\UserGpuPreferences` with `GpuPreference=2;` resolved the world-load crash on this machine. After that override, the client loaded into the world successfully while still connected to `127.0.0.1:14021`.
- World-stream framing status: the successful in-world `14021` capture is not one frame per `recv()` chunk. `emulator.protocol.framing.split_length_prefixed_frames` confirmed that each message starts with a little-endian `u16` body size and the true frame size is `declared_size + 2`. Running `emulator.tools.frame_index` against `captures/20260418-172702-runner` converted `2239` raw TCP chunks into `6254` protocol frames across `1228` distinct frame families.
- High-signal world-frame families: `captures/20260418-172702-runner/frame-index/summary.md` now shows the dominant repeated outbound frames for the working in-world session. The biggest families are size `40` / prefix `260095c289e0d06b` at `704` frames, size `31` / prefix `1d0049c1d234c9b2` at `355` frames with coordinate-like fragments such as `67860 67860`, size `35` / prefix `210090e6461339ba` at `241` frames, size `31` / prefix `1d00f67be12c7d6a` at `209` frames with fragments such as `9: 35263 35265`, and size `92` / prefix `5a0077ea8422d608` at `98` frames with the fragment `3 demigod`. These families are the best next targets for labeling and controlled replay experiments.
- World-family labeling status: `emulator.tools.world_family_labels` now labels repeated frame families heuristically and writes results to `captures/20260418-172702-runner/world-family-labels/summary.md`. The first pass labeled `582` of `1228` frame families. The clearest high-confidence labels are: `1d0049c1d234c9b2` and `1d00b80f09167a6a` as likely coordinate-pair payloads, `1d00f67be12c7d6a` as an indexed coordinate-pair payload, `1d004d23c18ad027` and `1d00b45f47d593c6` as coordinate-quad payloads that often end with `0 0`, `0e000040c575b0ed` and `0900000000000000` as likely keepalive or ack frames, `5a0077ea8422d608` as a named text-bearing payload because it carries `3 demigod`, and `260095c289e0d06b` / `150051baf4aa51e5` as tagged numeric-token families. This is enough to replace the manual top-family notes with a repeatable labeling pass.
